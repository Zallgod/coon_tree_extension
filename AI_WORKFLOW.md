## CLI TOOLKIT

These are the primary commands used during development.

ai-start
Initializes an AI dev session. Automatically generates architecture and debugging context.

doc
Prints the core architecture documents so they can be pasted into an AI chat.

search-code
Searches the repository for specific code references.

repo-diff
Shows file changes and code differences before committing.

## STANDARD SESSION WORKFLOW

1. Enter project directory

cd C:\Quant_Stack\coon_tree\extension

(or other project directory)

---

2. Initialize AI session

ai-start

This generates:

repo_map.md
repo_summary.txt
architecture_scan.txt
state_flow.txt
bug_scan.txt

These files summarize the repository structure and behavior.

---

3. Provide architecture context to AI

Run:

doc

Paste the output into the AI conversation.

This provides the AI with:

repo_map.md
invariants.md
data_models.md
engine_flow.md
repo_summary.txt

These files allow the AI to understand the project without scanning the entire repository.

---

4. Investigate code

Use targeted search:

search-code <keyword>

Examples:

search-code chrome.tabs
search-code stateManager
search-code apply
search-code persistence

This helps identify where functionality exists before asking AI to modify it.

---

5. Review changes before committing

Run:

repo-diff

This shows:

• modified files
• exact code changes

Use this to confirm that AI-generated edits did not unintentionally modify unrelated code.

## ARCHITECTURE FILES

These files define the system structure and must be respected by AI tools.

repo_map.md
List of files and responsibilities.

invariants.md
Rules the architecture must never violate.

data_models.md
Definitions of core objects and structures.

engine_flow.md
How events and mutations move through the system.

repo_summary.txt
Compact architecture overview generated by CLI tools.

---

## CORE ENGINE RULES

All state mutations must go through:

stateManager.apply()

chromeSync translates Chrome events into engine mutations.

chromeId is a transient runtime binding.

nodeId is the permanent identity of a node.

UI code must never mutate engine state directly.

Persistence code may read/write storage but must not mutate engine state.

---

## AI PROMPTING RULES

Avoid large rewrite prompts.

Do NOT ask AI to:

"rewrite this file"
"refactor everything"

Instead use targeted instructions such as:

"Modify only this function."

"Do not change other logic."

"Respect invariants.md."

Small targeted edits produce far more reliable results.

---

## SESSION GOAL

Use architecture documents and targeted searches to give AI minimal but accurate context.

This reduces token usage and prevents hallucinated code changes.
